How to Size a Scale Computing Cluster

A from-scratch walkthrough: what each number means, every formula explained term by term, and a worked example at each step. No prior infrastructure knowledge assumed.

"Sizing" means working out exactly how much hardware a customer needs to run their software — not too little (it falls over), not too much (they overpay). This guide builds that skill from the ground up. We'll define every word before we use it in a formula, and every formula comes with a plain-English breakdown and an example you can follow on a calculator.

Start here — the vocabulary

Every formula later is just these building blocks combined. Read this once and the rest will click.

Server / Node
A single physical computer in a rack. In a Scale Computing cluster we call each server a node. A cluster is built from several identical nodes.
Cluster
A group of nodes that work together as one system. If one node fails, the others keep the workload running. The whole point of sizing is deciding how many nodes and how big each one needs to be.
HCI (Hyper-Converged Infrastructure)
An architecture where every node provides compute (CPU + memory) and storage at the same time, instead of having separate servers and a separate storage array. Scale Computing HyperCore is HCI. "HCI node" just means a node that runs virtual machines.
CPU, Core, Thread, Socket
The CPU is the processor chip that does the computing. Each CPU contains several cores — independent worker units; a "24-core" CPU has 24 of them. A socket is the slot a CPU plugs into; a "dual-socket" server holds 2 CPU chips. A thread is a logical worker: with Intel Hyper-Threading, each core presents as 2 threads, so a 24-core CPU shows 48 threads. Cores are what we size and license against; threads matter for fitting a single big VM.
GHz (gigahertz — clock speed)
How fast a core runs, in billions of cycles per second. A 3.0 GHz core does 3 billion ticks a second. More GHz = faster per core, but (as we'll see in §11) GHz alone doesn't tell you how much work a core does.
RAM (memory)
Fast, temporary working memory, measured in gigabytes (GB). Every running program and VM holds its live data here. Unlike storage, RAM can't be safely over-allocated — if a VM needs 16 GB, it really needs 16 GB.
Storage (disk capacity)
Permanent space for files and virtual machines, on drives (HDD = spinning hard disk, SSD/NVMe = flash). Measured in terabytes (TB).
VM (Virtual Machine)
A complete computer running in software on top of a node — its own OS, apps, CPU, and memory. One physical node runs many VMs at once. The customer's "workload" is just their collection of VMs.
vCPU (virtual CPU)
A virtual core handed to a VM. A VM configured with "4 vCPUs" believes it has a 4-core processor. The clever part: because most VMs are idle most of the time, you can hand out more vCPUs than you have physical cores — this is overcommitment (see §4).
Raw vs. Usable capacity
Raw = the total capacity printed on the drives. Usable = what's actually available to VMs after the system keeps copies for safety and holds some back for repairs. Usable is always smaller than raw — and the gap is not waste, it's protection.
Provisioned vs. Used
Provisioned = how much was promised/allocated to the VMs. Used = how much is actually consumed right now. They differ because of thin provisioning — allocating a 100 GB disk to a VM that has only written 12 GB. You size to used, because that's the real demand.
IOPS (I-O-P-S)
Input/Output Operations Per Second — how many individual read or write requests the storage can satisfy each second. It's a measure of storage speed, separate from storage capacity. A database hammering small reads needs high IOPS even if it stores little data.
Snapshot
A point-in-time state of a VM that stores only the changes since it was taken (a "delta"), so it depends entirely on the original VM. It's a quick-rollback tool — not a standalone backup (see §3). Keeping snapshots consumes extra storage, so we reserve room for them.
N-1 ("N minus one")
The redundancy rule: design the cluster so it still runs the entire workload with one node removed (failed or being updated). "N" is the node count; "N-1" is what's left when one is gone. This is why you always buy a bit more than you strictly need.
Hold this picture: a cluster is a small team of identical servers (nodes). Each runs many virtual computers (VMs). You're deciding how many team members to hire and how strong each must be — so the team still finishes the job even if one member calls in sick (N-1).

1Gather the inputs

You can't size what you haven't measured. So the first job is to capture what the existing environment really runs — usually from a discovery scan (a Live Optics or RVTools report, for example). Here's what each measurement is and why it matters:

InputWhat it is, in plain terms
Total vCPUsAdd up the vCPUs across every VM. If 50 VMs each have 4 vCPUs, that's 200 vCPUs of demand. This is the raw compute appetite.
Used RAM
(not provisioned)		How much memory the VMs actually consume. We size to this, not to the larger "provisioned" figure, which is just the ceiling they were allowed.
Datastore used
(not provisioned)		How many TB of storage are actually written. Thin provisioning means VMs are often "allocated" far more than they've used.
Total & active VMs"Active" = powered-on. A powered-off VM uses storage but no CPU/RAM, so active VMs drive the density check (§9).
Largest single VM
(its vCPUs & RAM)		Crucial: one VM runs entirely on one node — it can't be split across nodes. So the biggest VM must fit inside a single node's usable capacity.
IOPS — avg, peak, P95Storage speed demand. P95 means "the 95th-percentile" — the level the workload stays under 95% of the time. We aim at P95 rather than the rare peak, so we don't over-buy for a once-a-month spike.
Network throughput & NIC speedHow much data moves in and out of the hosts today, plus the speed of the existing NICs (e.g. 10 or 25 GbE). This sets the east-west bandwidth the new cluster has to carry (§8) — and flags whether the current links are already a bottleneck.
Source CPUs
(model & socket count)		Which processors the customer runs today. Used to compare raw throughput, not just core counts (§11).
The golden rule of inputsSize to used and active — never to provisioned and total. Provisioned numbers are promises; powered-off VMs are dormant. Sizing to them buys hardware nobody is using.
ExampleA scan reports 76 VMs, 59 powered on, totalling 148 vCPUs, 423 GB used RAM (but 600 GB provisioned), and 8.2 TB datastore used (but 18 TB provisioned). We carry forward 148, 423 GB, and 8.2 TB — the used/active figures. The provisioned numbers just tell us the customer over-allocated, which is normal.

Know what you're sizing — workload archetypes

Before crunching numbers, recognise the kind of workload in front of you. Different applications lean on different resources, and knowing the shape tells you where to focus (and which advanced sections matter):

2Ground rules & reservations

Here's the idea beginners miss most: a node that says "96 cores, 768 GB RAM, 80 TB of drives" never gives all of that to VMs. The system reserves some for itself, keeps safety copies, and holds capacity back for a node failure. Let's go through each reservation and why it exists.

The operating system takes its cut

Every node runs the HyperCore OS, which needs some cores and RAM to do its job (running the hypervisor and the storage layer, called SCRIBE).

OS cores

1 core per node is set aside for the OS before counting cores available to VMs.

OS RAM (tiered)

SCRIBE uses more memory as you add drives, so OS RAM scales with drive-bay count:
4 GB (1–4 bays) · 8 GB (5–12) · 16 GB (13+).

Safety buffer

An extra 2 GB per node of breathing room. So total usable-RAM overhead = OS RAM + 2 GB.

ExampleA node with 384 GB RAM and 10 drive bays reserves 8 GB (OS, 5–12 bay tier) + 2 GB (buffer) = 10 GB. VMs get 374 GB, not 384.

Data is mirrored — RF2

RF2 means Replication Factor 2: every block of data is written to two different nodes. If a node dies, a full copy of everything it held still exists elsewhere, so nothing is lost.

Think of RF2 as always keeping a photocopy of every document on a colleague's desk. Twice as safe — but you need twice the filing space. That's why usable storage is roughly raw ÷ 2.

A capacity reservation — for the cluster's own use

The cluster keeps a slice of capacity for itself — the OS, logging, metadata and similar internal overhead. Rather than guess a fixed amount, the reservation is sized to the single biggest disk in the cluster and subtracted once from the cluster's total raw capacity — not once per node. Reserving the largest disk is deliberately conservative, so the reservation is never too small whatever drives are fitted.

Survive a node loss — N-1

This is the big one. The cluster must run the whole workload even with one node gone. So when we check "does CPU fit?" or "does RAM fit?", we check it against the capacity of all nodes minus one.

A relay team of 4 that can still finish the race if one runner pulls up. You don't staff for the best case — you staff so a single absence doesn't lose the race.
Why storage is different at N-1Because RF2 already keeps a second copy of all data, losing a node doesn't shrink your usable storage — the surviving copy covers it. So storage capacity is the same at N-1 as at full cluster; only CPU and RAM need the one-node-down headroom.

Cluster size limits

Minimum 2 HCI nodes

You need at least 2 VM-running nodes per cluster, or there's nothing to fail over to.

Maximum 8 nodes

Past 8 nodes, a build splits into multiple clusters (each its own manageable unit).

Hybrid flash rules

On mixed flash+HDD nodes: flash must be 7%–25% of raw capacity, with at least 3 HDDs per flash disk so the slow tier can absorb data pushed down from flash.

3Growth, snapshots & day-one caps

A customer keeps hardware for years, and their workload grows the whole time. So you don't size for today — you size for the horizon (default 5 years). Three adjustments turn today's demand into the future demand the cluster must hold.

Compound growth

If a workload grows 10% a year, year 2 grows on top of year 1's bigger number — like compound interest. Over 5 years that's not +50%, it's ×1.61.

growth_factor = (1 + growth_rate) ^ years = (1 + 0.10) ^ 5 = 1.10 × 1.10 × 1.10 × 1.10 × 1.10 ≈ 1.61 # today's demand × 1.61 = 5-year demand
growth_rate
The yearly growth as a decimal. 10% per year is 0.10. (The "^" symbol means "to the power of" — multiply by itself that many times.)
years
The planning horizon, default 5.
growth_factor
The multiplier you apply to every "today" number to get the "5-years-from-now" number.

Snapshot overhead

On top of growth, storage carries extra weight for snapshots. The default reserve is 20% more storage. CPU and RAM don't get a snapshot reserve — only storage does, because a snapshot is stored data.

A snapshot is not a backupA snapshot captures only the changes since it was taken — a "delta" — so it depends entirely on the original VM and its data. Delete the VM and its snapshots become worthless. On a single cluster, a snapshot is a dependent point-in-time copy: great for quick rollback, but no protection against losing the VM or the cluster.

HyperCore does use snapshots for cluster-to-cluster replication: with two clusters (production and DR) replicating to each other, the full VM and its snapshots live on both sites — and that pair can be treated as a rudimentary backup. The independence comes from the second cluster, not from the snapshot itself.
projected_vcpus = today_vcpus × growth_factor projected_ram = today_ram × growth_factor projected_storage = today_storage × growth_factor × (1 + 0.20) # the ×1.20 is snapshots

Day-one caps — don't ship nearly full

Even after planning for growth, we don't want a brand-new cluster to arrive already 90% full. So two caps reserve extra headroom for today:

CapDefaultMeasured againstIn plain terms
Max day-one storage use50%Full-cluster usableOn the day it ships, today's data should fill at most half the usable storage — leaving the other half for the 5 years of growth.
Max day-one RAM use50%N-1 available RAMToday's memory should use at most half of what's available even with a node down — so there's room as it grows.

If today's demand would already exceed a cap, you size up — add a node, or choose bigger components — until it fits under the cap. How much day-one headroom to reserve is a judgement call: reserve more and the cluster is roomier but costs more; reserve less and it ships closer to full on day one.

4Sizing CPU

CPU sizing answers: how many physical cores, across how many nodes, do we need to satisfy all those vCPUs — even with a node down?

The key idea: overcommitment

Real VMs rarely use all their vCPUs at once — a 4-vCPU VM might average 10% busy. So you can safely hand out more virtual cores than you have physical ones. The vCPU:core ratio (default 3.0) says how aggressively: at 3:1, each physical core backs 3 vCPUs.

Like a gym selling more memberships than it has treadmills, or a restaurant taking more reservations than tables — it works because not everyone shows up at once. Set the ratio too high and you get a queue (slow VMs); too low and you've bought treadmills nobody uses.
# Step 1: how many physical cores does each node actually offer to VMs? effective_cores = p_cores × w_pcore + e_cores × w_ecore # normally just the core count usable_cores_per_node = effective_cores − 1 # minus the OS core # Step 2: how many cores does the workload demand? required_cores = ceil( projected_vcpus / vcpu_ratio ) # round UP to a whole core # Step 3: the fit test — must pass with ONE node missing usable_cores_per_node × (number_of_nodes − 1) ≥ required_cores
p_cores / e_cores
On modern Intel chips, "P-cores" are big performance cores and "E-cores" are small efficiency cores. On normal server CPUs they're all P-cores. (More in §11.)
w_pcore / w_ecore
Weights — how much each core type counts. Defaults 1.0 and 0.0: a P-core counts fully, an E-core counts as zero (because licensing only covers P-cores today). For a normal CPU this just gives back its core count.
effective_cores
The core count we actually size and license against, after that weighting.
− 1 (OS core)
One core per node handed to the operating system, leaving the rest for VMs.
vcpu_ratio
The overcommit ratio, default 3.0.
ceil( )
"Round up." You can't buy 79.3 cores, so 79.3 becomes 80.
required_cores
The physical cores the workload needs once overcommit is applied.
(number_of_nodes − 1)
N-1 again — we test against all nodes but one, so a single failure is survivable.
ExampleThe 5-year projected demand is 238 vCPUs. At the 3:1 ratio: 238 ÷ 3 = 79.3 → round up to 80 cores required. Each node is a dual-socket box with 48 physical cores; minus 1 for the OS gives 47 usable cores per node. Can 2 nodes (that's 3 nodes minus one, for N-1) cover it? 47 × 2 = 94 usable cores ≥ 80 ✓ — with 94/80 = 17.5% headroom to spare. So 3 nodes.
Two safety checks on topLargest-VM fit: since a VM can't span nodes, the biggest VM's vCPUs must fit within one node's threads. A 64-vCPU VM can't live on a node that only offers 48 threads. GHz floor: if a high overcommit ratio would leave the new cluster with far less total GHz than the old environment, that's your cue to push back toward more or faster cores so you don't under-power them.

Advanced: fast cores vs. many cores

Once you know how many cores a workload needs, there's still a choice of what kind of CPU — many cores at a moderate clock, or fewer cores at a high clock. The right answer depends on the shape of the workload, and the measurements you gathered in §1 will tell you which one you're holding.

The tell-tale — saturated cores. Don't just look at the average CPU figure; look at the individual cores. If overall usage seems modest but a few cores sit pinned near 100% while the rest idle, that's the fingerprint of a single-threaded application — a program that can only run on one core at a time. (Live Optics flags this directly in its dashboard; in an RVTools-style export the clue is a VM whose max CPU hits 100% even while its average stays low.)

A single-threaded process gets no benefit from extra cores — it physically can't spread its work across them. The only thing that speeds it up is a faster core: high clock speed, the all-core turbo we met in §11. So a workload like that wants a CPU with a high clock even at the cost of a lower core count. Throwing more cores at a single-threaded bottleneck just buys capacity the application will never touch.

One chef who only works the grill won't cook any faster because you added twenty more stations — they can still only stand at one. A hotter, faster grill is what speeds them up. A restaurant with twenty chefs, on the other hand, loves twenty stations.

The opposite case — consolidation hosts. A node packed with many independent VMs is the mirror image. Each VM is its own small workload, so more physical cores means more real slots to run them side by side, and clock speed matters less. There's also a second reason cores win here, tied straight back to the overcommit ratio: with fewer cores you have to push the vCPU:core ratio higher to fit the same vCPUs — and past a point, that backfires.

When the ratio gets too highRemember the CPU scheduler hands each vCPU its turn on a physical core. Overcommit modestly and it juggles smoothly. Overcommit too hard and VMs start spending real time waiting for a free core (often called "CPU ready time"), and bigger multi-vCPU VMs stall waiting for several cores to come free at the same moment. Beyond that point the scheduling overhead does more harm than the consolidation does good. More physical cores let you keep the ratio sane and the scheduler out of trouble.

The takeaway: let the measured shape steer the CPU choice. Saturated cores and single-threaded bottlenecks → favour fewer, faster cores (high clock), and don't try to fix them with core count. Broad, many-VM consolidation → favour higher core counts, which keep the overcommit ratio low and the scheduler happy.

5Sizing RAM

RAM is simpler than CPU because you can't overcommit it safely — if VMs need 400 GB of memory, you must have 400 GB. The only twist is subtracting the OS reservation, and making sure the biggest VM fits on one node.

usable_ram_per_node = node_ram − ( os_ram_for_bays + 2 GB buffer ) # Must cover projected RAM with a node down, AND host the largest single VM usable_ram_per_node × (number_of_nodes − 1) ≥ projected_ram usable_ram_per_node ≥ largest_VM_ram
node_ram
The physical RAM installed in one node, e.g. 384 GB.
os_ram_for_bays
The tiered OS reservation from §2 (4 / 8 / 16 GB depending on drive bays).
usable_ram_per_node
What's left for VMs after the OS and buffer.
largest_VM_ram
The memory of the single biggest VM — it must fit on one node, since VMs don't split.
ExampleProjected RAM need is 680 GB, and we're checking a 3-node cluster (so 2 nodes at N-1). Each node has 384 GB; usable after a 10 GB overhead is 374 GB. Two nodes give 374 × 2 = 748 GB ≥ 680 ✓. And the largest VM (say 96 GB) easily fits in 374 GB ✓. RAM is fine on 3 nodes.

Note: any node RAM option that can't hold the largest VM after overhead is thrown out before sizing even starts — there's no point offering a node that can't host the biggest guest.

Advanced: keeping memory local (NUMA)

This one is a finer point — virtualization tuning rather than basic sizing. You can build a perfectly good cluster without ever thinking about it, but in high-performance, latency-sensitive environments it's a real win when you get it right.

A node with two physical CPUs doesn't have one big shared pool of RAM. Each CPU has its own memory banks wired directly to it. When a CPU reaches its own memory it's fast (local); when it has to reach the other CPU's memory, the request crosses the link between the two chips, which is slower (remote). This split is called NUMA — Non-Uniform Memory Access — and each "CPU plus the memory attached to it" is one NUMA node.

Picture a two-room office where each worker keeps a filing cabinet right beside their desk. Grabbing a file from your own cabinet is instant; walking to the other room for a colleague's cabinet costs time. You'd rather keep each person's work in their own cabinet — and you'd rather not give one person more files than their own cabinet can hold.

The sizing implication: wherever you can, a single VM's memory should fit inside one CPU's share of the node — roughly node_RAM ÷ number_of_sockets. If a VM needs more than that, it spills across both NUMA nodes, and every time it touches the far memory it pays the remote-access penalty. In the worst cases the hypervisor ends up copying or shuffling memory to cope, so a badly-placed large VM can quietly waste capacity and speed at the same time.

When to botherFor ordinary mixed workloads, don't lose sleep over it — the hypervisor places VMs sensibly on its own. But when you're sizing for databases, real-time systems, or anything chasing every microsecond, check that the big VMs fit within a single socket's worth of memory. If a critical 200 GB VM is destined for a node with only 128 GB per CPU, that's your cue to pick a node with more RAM per socket (or to plan fewer, larger VMs per node) — so the workload stays local and fast.

6Sizing storage

Storage sizing turns the drives' raw capacity into usable capacity by applying the two reservations from §2 — the cluster reservation and RF2 mirroring — then checks it covers the projected need. Both are applied at the cluster level, not per node.

# Per CLUSTER: total raw of every disk, minus ONE biggest-disk reservation, # then halved for RF2 mirroring. cluster_raw = raw_capacity_per_node × nodes_in_cluster usable = ( cluster_raw − biggest_disk ) / 2 cluster_usable = sum of usable across clusters # usually just one cluster # Must cover projected storage and stay under the 50% day-one cap cluster_usable ≥ projected_storage today_used ≤ 0.50 × cluster_usable
cluster_raw
The printed size of every drive in the whole cluster, added together. Three nodes of ten 1.92 TB drives = 3 × 19.2 = 57.6 TB.
− biggest_disk
A single reservation per cluster (OS, logging, metadata), sized to the largest disk so it's never an under-reservation. Subtracted once — not once per node.
/ 2
RF2: every block is mirrored to a second node, so usable is half of what remains.
cluster_usable
Add up each cluster's usable figure (most sizings are a single cluster). Same value at N-1 — RF2 already covers a node loss.
ExampleThree nodes, each with ten 1.92 TB NVMe drives. Total cluster raw = 3 × 19.2 = 57.6 TB. Subtract one biggest disk (1.92 TB) for the cluster reservation → 55.68 TB. Halve for RF2 → 27.84 TB usable. Projected need ~16 TB fits, and today's 8.2 TB is under the 50% cap (50% of 27.84 ≈ 13.9 TB). ✓

7Sizing IOPS

An IO is one storage request (read a block, write a block). IOPS is how many of those per second the storage can handle — its speed. Drives have a rated IOPS number, but the workload never sees all of it: the storage software takes overhead, and mirroring multiplies writes. So when you size IOPS, you have to account for both losses.

# Writes get multiplied by replication; reads don't. This is "write amplification". write_amp = read_frac + (1 − read_frac) × RF = 0.70 + (1 − 0.70) × 2 = 0.70 + 0.60 = 1.30 raw_iops_per_node = sum of (drive_count × each drive's rated IOPS) derated_iops_per_node = raw_iops_per_node × (1 − 0.35) # 35% software overhead net_iops_per_node = derated_iops_per_node / write_amp cluster_net_iops = net_iops_per_node × number_of_nodes n1_net_iops = net_iops_per_node × (number_of_nodes − 1) # speed with a node down
read_frac
The fraction of IO that is reads, default 0.70 (70% reads, 30% writes — typical mixed workload).
RF
Replication factor, 2. Each write happens on 2 nodes, so writes count double.
write_amp
The average multiplier on IO once you account for mirrored writes. Reads stay 1×, writes become 2×; blended at 70/30 that's 1.30×.
× (1 − 0.35)
A 35% "derating" for the storage stack (SCRIBE) overhead — you keep 65% of the drives' rated IOPS.
net_iops_per_node
The real, usable IOPS one node delivers to workloads.
Then compare to demandThe net available IOPS is checked against the workload's measured P95 demand. Per-drive IOPS values are deliberately a tunable knob, because vendor-published drive numbers are optimistic — we trust the measured workload over the spec sheet.
ExampleSuppose a node's drives are rated at 850,000 raw IOPS. Derated: 850,000 × 0.65 = 552,500. After write-amp: 552,500 ÷ 1.30 ≈ 425,000 net IOPS/node. Across 3 nodes ≈ 1.27M; even at N-1 (2 nodes) ≈ 850k — versus a measured P95 demand of ~16,000. IOPS is nowhere near the constraint here, which is common for all-flash.

8Sizing the network

Here's a resource beginners almost always forget: the network. In a hyper-converged cluster the storage isn't a separate box — it lives across the nodes, and the nodes constantly keep each other's copies in sync over the network. So the network is part of the storage system. Undersize it and a cluster full of fast flash will still crawl.

Why HCI leans so hard on the network. Remember RF2 from §2: every write is mirrored to a second node. That mirror copy doesn't travel by magic — it crosses the network between nodes. Traffic between nodes is called east-west; traffic in and out of the cluster (users reaching their VMs) is north-south. RF2 makes east-west the heavy one. Add node rebuilds after a failure, VM live-migrations, and replication to a DR site, and the inter-node network carries a lot.

If the disks are warehouses, the network is the road between them. The fastest warehouses in the world don't help if a single country lane connects them — the trucks queue, and everything waits on the road.

What you actually size. First, the link speed of each port. The supported speeds depend on the node type:

Edge nodes

1 or 2.5 GbE (Base-T) or 10 GbE (SFP+).

Rack nodes

10 GbE (Base-T or SFP28) or 25 GbE (SFP28).

Base-T is ordinary RJ45 copper cabling; SFP+ / SFP28 are slots that take a fibre or direct-attach transceiver, used for the faster links. The busier the writes and the bigger the cluster, the faster the link should be — all-flash and high-IOPS workloads push the most east-west traffic and earn the fastest available. Second, the number of NICs per node, which on a Scale node is always 1, 2 or 4 — and that choice decides your redundancy and whether the storage backplane shares the wire with the LAN or gets its own:

NICs / nodeLAN & backplaneRedundancySwitch ports
1Both ride one port; backplane kept apart by a VLAN tagNone — the port is a single point of failureHybrid (tagged backplane + untagged LAN)
2Still shared (backplane over VLAN), but active/passive — one live, one on standbyYes — the standby takes over on a faultHybrid
4Physically separated: a LAN pair plus a dedicated backplane pair, each active/passiveYes — on both LAN and backplaneLAN pair hybrid (VM VLANs + untagged management); backplane pair untagged

The redundancy model is active/passive throughout — a standby port takes over on failure rather than both running at once. Extra NICs beyond these can be added, but for now they act as additional passive LAN ports. The sizing takeaway: more NICs buy redundancy, and at four you get a dedicated backplane that keeps heavy storage replication off the LAN — the right call for busy, all-flash clusters.

The classic mistakeSpeccing big, fast NVMe drives behind a slow or single-port network. The drives can serve hundreds of thousands of IOPS, but every mirrored write still has to fit down the link to the partner node — so the network, not the disk, becomes the ceiling. Match the network to the storage's appetite, not the other way round.

9N-1 & the determinant

You've now sized four things independently — CPU, RAM, storage, IOPS — and each one suggests a node count at N-1. The cluster needs the largest of those. Whichever resource forces that number is called the determinant — the bottleneck that decided the size.

A chain is only as strong as its weakest link. If CPU needs 3 nodes but RAM needs 2 and storage needs 2, you build 3 — and CPU is the determinant.

Naming the determinant is how you explain a sizing in one sentence — state the binding resource, what it needed, what's available at N-1, and the headroom between them:

How to state it"Determined by CPU — 80 cores required vs 94 usable cores available at N-1 (17.5% headroom)." That's just the §4 result, narrated. If anyone questions the size, this is the sentence you defend it with.
Watch VM density tooNode count isn't only about CPU, RAM and storage — keep an eye on how many VMs land on each node. As a rule of thumb, past roughly 100 active VMs on a single node, scheduling and management get harder, and a single node loss takes more VMs down with it. If a design packs them that tightly, consider spreading across more nodes.

10Small clusters, quorum & single-node systems

N-1 assumes you have a cluster of several nodes. The edges of that assumption — very small clusters, and a single node on its own — have their own rules worth understanding.

Why a cluster needs a tie-breaker (quorum)

Picture a 2-node cluster where the link between the two nodes breaks, but both nodes are still alive and running. Each one now thinks "the other died — I'd better take over." If both act on that, you end up with two copies of every VM running independently and writing different data — a corrupting mess called split-brain.

The fix is quorum: a majority vote. A cluster only stays active if it can see more than half the votes. With an odd number of nodes there's always a majority. With an even number (like 2) you add a tiny external tie-breaker called a witness — a lightweight service (a small VM or a cloud endpoint) that holds the deciding vote, so exactly one side stays up and the other steps back. No drama, no split-brain.

Three people can always settle a vote two-to-one. Two people deadlock — so you bring in a referee (the witness) whose only job is to cast the tie-breaking vote.

Storage-only nodes

Sometimes a workload needs lots of capacity but not much compute. Rather than buy full VM-running nodes just for their disks, you can add storage-only nodes — nodes that join the cluster's storage pool but run no VMs. They add usable TB and IOPS without adding licensed compute cores: handy for capacity-heavy, compute-light designs.

Single-node systems (SNS) — not a cluster at all

At the far edge of "small" is a single-node system (SNS): one node, on its own. Common at the edge — a retail store, a branch office, a remote site — where a full cluster is overkill.

An SNS has no safety net — so backup or replication is mandatoryAn SNS is not a cluster. There's no second node, so there's no N-1, no failover, no "the others carry on." If that one node dies, everything on it is down until it's repaired. That makes off-box backup and/or replication a requirement, not an option — typically replicating to another SNS or cluster at a second site (which, as with snapshots in §3, is also where you finally get a real, independent copy of the data).

Sizing an SNS follows the same storage rules as a cluster, with one exception: you don't subtract the biggest disk for the cluster reservation. On a single node that reservation would cost a painfully large share of the total — too expensive to justify — so it's dropped. And one special case is worth committing to memory:

One disk: raw = netWith multiple disks an SNS still mirrors data across them (RF2), so usable ≈ raw ÷ 2. But an SNS with a single disk has nothing to mirror to — there's no second disk to hold the copy. So there's no halving and no reservation: raw capacity is the usable capacity (RAW = NET). That single disk also means zero local redundancy, which is exactly why the backup/replication rule above is non-negotiable.

11Benchmark intelligence — why core count alone lies

Here's a trap: an old server with 40 cores is not equal to a new server with 40 cores. Modern cores do far more work per clock tick — this is called IPC (Instructions Per Clock). Counting cores alone makes you buy a huge modern cluster just to "match" an old core count you don't actually need to match.

Old trucks vs. new trucks: ten old trucks might move the same freight as four new ones. If you only counted "trucks," you'd massively over-buy. You need to compare how much each truck actually hauls — that's a benchmark.

Step one: size each CPU honestly

IdeaWhat it means & how it's handled
Sizing clockA CPU has several speeds. Server CPUs are sized on their all-core turbo — the speed they sustain with every core busy, in a cooled datacenter. Laptop-class chips are sized on their slower base clock, because their headline turbo can't be held for long on thin edge hardware.
P-cores vs E-coresModern Intel chips mix big "performance" and small "efficiency" cores. We store the true total but size on P-cores × 1.0 + E-cores × 0.0 — E-cores count as zero because the software is licensed per P-core today.
Performance indexA single throughput score on the SPECrate2017 scale (an industry CPU benchmark). Server chips use their native SPECrate score; desktop chips are converted from the PassMark benchmark by multiplying by 0.00386 (an approximate bridge between the two scales, good to about ±20%).

Step two: compare old vs. new

# Total throughput the customer runs today source_throughput = sum over each source CPU of ( per-socket score × number of sockets ) # Total throughput the recommended cluster delivers target_throughput = per-node CPU score × number of VM-running nodes ratio = target_throughput / source_throughput
per-socket score
The SPECrate value for one CPU chip. (A PassMark score is converted with ×0.00386 first.)
number of sockets
How many of that CPU the customer has. Six dual-socket servers = 12 sockets of that chip.
ratio
How many times more compute the new cluster delivers. 4.8 means "about 4.8× the throughput of what they run now."
ExampleThe customer runs 6 sockets of an old Xeon Silver 4110 (SPECrate ≈ 37 each): 37 × 6 ≈ 222 total. The recommended 3-node cluster's CPUs total ≈ 1071. Ratio = 1071 ÷ 222 ≈ 4.8×. So you can tell the customer the new cluster delivers roughly 4.8× the throughput of what they run today — a fair comparison, not a misleading core count.
Two ways to use the comparisonYou can treat throughput as context — size on cores and clock as usual, and show the benchmark gain as reassurance the new kit is amply powerful. Or you can treat it as a rule — require the new cluster to deliver at least the old environment's throughput (grown to the horizon) before it counts as a fit. The rule is powerful, but lean on it carefully: the SPECrate↔PassMark bridge is only ~20% accurate, so check it against a few known-good sizings before trusting it to set node counts.

12Picking the winner

Usually several clusters "fit." Don't just grab the one with the fewest nodes — weigh every option and pick the best overall. The way to do that is to score each candidate with a weighted sum of the trade-offs that matter, where a lower score is better. Three forces pull on it:

score = cost×fleet_cost + license×total_cores + waste×over_provisioning
ForceIn plain termsDefault weight
Fleet costTotal hardware spend = number of nodes × (the model's price + a fixed per-node overhead for switches, rack space, power, ops).1.0
Core licensingSoftware you pay for per CPU core (HyperCore, plus Windows/SQL/Oracle). Fewer total cores = lower recurring cost.1.5
Wasted capacityBuying far more CPU/RAM/storage than needed. Capped per resource so one wildly oversized dimension can't dominate the decision.50

The lesson: good sizing doesn't blindly minimize node count. A slightly bigger cluster can be the better answer when it's much cheaper to license and more tightly fitted. You're always balancing "does it fit" against "is it sensible to buy and run."

13Full worked example

The starting point (from a discovery scan of the existing environment): 148 vCPUs · 423 GB used RAM · 8.2 TB datastore used · 59 active VMs · ~16,000 P95 IOPS · today's CPUs = 6 sockets of Xeon Silver 4110.

1 · Project 5 years (10%/yr, 20% snapshots): growth_factor ≈ 1.61 → ~238 vCPUs, ~680 GB RAM, ~16 TB storage.

2 · CPU: 238 ÷ 3.0 = 79.3 → 80 cores required. A 48-core node gives 47 usable; at N-1, two nodes = 94 ≥ 80 (17.5% headroom) → 3 nodes.

3 · RAM: 680 ÷ 2 = 340 GB needed per node at N-1; a 384 GB node (374 usable) clears it.

4 · Storage: 3 nodes × ten 1.92 TB NVMe = 57.6 TB cluster raw; minus one biggest disk, ÷ 2 → (57.6 − 1.92) ÷ 2 = 27.84 TB usable, above 16 TB and under the 50% day-one cap.

5 · IOPS: ~425k net/node × 2 (N-1) ≈ 850k versus 16k demanded — not even close to a limit.

6 · The determinant is CPU — it forced the 3-node count; RAM, storage and IOPS all fit comfortably inside it.

7 · Benchmark: new ≈ 1071 vs old ≈ 222 → ~4.8× the throughput. Final answer: a 3-node all-flash cluster, CPU-bound, with five years of growth and a node-failure cushion already built in — and nearly 5× the compute of the estate it replaces.

14Sanity-check your answer

Whether you sized by hand or a tool did the arithmetic, never hand over a result you haven't sanity-checked. This is the step that lets you catch a bad number before a customer does — and the whole reason to understand the method rather than just trust the output. Run down this list:

CheckWhat you're looking for
Did I size to used, not provisioned?The single most common error. If storage or RAM looks huge, make sure you sized to real consumption, not the thin-provisioned ceiling.
Is there enough raw compute?Compare the new cluster's total GHz and benchmark throughput (§11) against the source. A "consolidated" cluster with far less compute than they run today is a red flag, not a win.
Is the headroom sensible?A few percent means no room for growth or a bad day. Triple the need means you over-bought. Aim for deliberate slack, not accidental.
Does the largest VM actually fit?Re-check the biggest VM against one node's cores, threads and RAM — and ideally one NUMA domain (§5). A VM that can't be placed sinks the whole design.
Did I forget the network?Fast disks behind a slow link (§8) is a silent bottleneck. Confirm the NICs match the storage's appetite.
What's the determinant — and does it make sense?Know which resource set the node count (§9). A database that came out "storage-bound" with idle CPU, or a web tier that came out "capacity-bound," is a sign something was mismeasured.
Is redundancy right for the site?Multi-node gets N-1; a single node (§10) gets none — so confirm backup/replication is in the plan wherever there's no failover.
The whole pointA sizing tool is a calculator, not an oracle. Understand the reasoning in this guide and its output becomes something you can read, question and correct — instead of a number you cross your fingers and quote.
